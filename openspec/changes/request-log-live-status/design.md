## Context

当前日志链路的真实行为是：代理入口先创建一条 in-progress `request_logs` 行，后续在成功、失败或部分异常路径中更新同一条记录；日志页则通过列表接口读取这些行，并用 `status_code` 是否为空来推导“还在进行中”。这套模式有两个直接体验断点：一是前端只能依赖手动刷新或定时轮询，因此状态变化看起来不够及时；二是流式响应开始后如果下游用户取消、网络断开或浏览器停止接收，日志并不总能在同一时刻收口成稳定终态，导致列表里长期出现“转圈未结束”的请求。

与此同时，部分统计逻辑依赖 `request_logs` 的终态数据。如果请求已经取消但日志仍停留在进行中，后续统计就会同时出现“进行中数量不准”和“成功类指标被脏数据影响”的体验问题。因此这次设计既要解决“更实时”，也要解决“更可信”。

## Goals / Non-Goals

**Goals:**

- 让用户在不手动刷新的前提下，于短时间内看到进行中请求的状态变化。
- 让下游取消、客户端断开、流式中断等场景在日志中稳定收口为终态，不再长期显示为进行中。
- 让长时间未收口的请求具备自动兜底收敛能力，避免列表与统计长期被脏状态污染。
- 让日志 UI 继续用状态码作为主状态表达，请求进行中只显示加载指示且不显示占位符 `-`，同时把上游返回的错误信息用错误色明确突出。
- 保持现有日志列表 API 为主数据源，尽量用向后兼容方式完成升级。

**Non-Goals:**

- 不在本次变更中把日志系统整体改造成完整事件溯源平台。
- 不引入 Redis、消息队列或外部实时基础设施。
- 不重做整个 Dashboard 的视觉体系，只覆盖请求日志与相关统计口径。
- 不改变调用方的错误协议格式，只改后台日志与管理端体验。

## Decisions

### Decision 1: 保留列表接口作为真相源，新增“轻量实时刷新链路”而不是直接推整表

**Choice:** 继续以 `GET /api/admin/logs` 返回的日志快照为真相源；新增一个轻量实时链路，只负责通知“哪些可见请求状态变了”或“当前页需要刷新”，而不是直接把完整分页数据改为流式主协议。

**Why:** 用户真正需要的是“状态尽快变对”，不是必须知道底层是轮询还是推送。保留现有列表接口，可以把筛选、分页、排序和兼容性风险控制在最小范围，同时显著降低实时化改造成本。

**Alternative considered:** 直接把整张日志表改为全量 SSE 推送或 WebSocket 同步。  
**Rejected because:** 当前列表有分页和筛选语义，直接推整表会显著增加一致性、补历史和重连复杂度，超出本次体验修复目标。

### Decision 2: 继续复用 `request_logs` 单行模型，但把“取消/中断”收口成明确终态

**Choice:** 不新增独立事件表作为实施前提；仍以单条 `request_logs` 记录承载一次请求的最终结果，并要求取消/中断场景必须写成终态。用户可见状态通过 `status_code`、失败原因、失败阶段和衍生状态统一映射得到。

**Why:** 现有数据模型已经覆盖大部分信息来源，真正缺的是“收口不完整”和“展示不友好”。先把终态写准，能用最小改动解决最大体验问题。

**Alternative considered:** 先引入 append-only 事件表，再由 UI 回放每个阶段。  
**Rejected because:** 这是更重的可观测性体系升级，适合作为后续增强，但不是本次用户体验止损的最短路径。

### Decision 3: 对“卡住的进行中请求”采用双层收敛策略

**Choice:** 第一层在请求生命周期内，凡是能检测到下游取消、客户端断开、流式中断的路径，都必须立刻更新对应日志行；第二层在日志查询和统计聚合前做 best-effort stale reconciliation，把超过阈值仍未收口的进行中请求自动改成终态。

**Why:** 仅依赖即时收口无法覆盖进程抖动、网络异常或异常路径漏更新；仅依赖事后对账又会让用户在操作时持续看到错误状态。双层策略可以兼顾即时体验和最终一致性。

**Alternative considered:** 只做读时修正，不补即时写入。  
**Rejected because:** 用户最在意的是“取消后马上别再转圈”，只做读时修正仍然会让实时体验显著滞后。

### Decision 4: 成功类性能统计只统计真正收口成功的请求

**Choice:** 对平均 TTFT、TPS、平均响应时间等“成功类性能指标”，只统计明确完成成功的请求；对已取消、已中断、异常补偿关闭的请求，只保留其终态信息，不混入成功类均值。

**Why:** 用户对这些指标的直觉是“系统成功完成请求时的表现”。如果把半途取消或未完成的请求混进去，指标会变得看起来很奇怪，也无法被信任。

**Alternative considered:** 只要有部分 TTFT/Token 数据就继续纳入性能均值。  
**Rejected because:** 被中途取消的流式请求不代表一次完整成功的服务过程，纳入后会扭曲用户对系统性能的理解。

### Decision 5: 日志主界面以“状态码 + 自动变化提示”为主，异常信息单独强调

**Choice:** UI 继续保留当前列表主体，但主状态区只保留状态码；请求进行中时只显示加载指示，不再额外显示状态文字或占位符 `-`；上游返回的错误信息统一用错误色强调。实时更新时对变化行做轻量提示，不要求用户先展开详情才能知道状态是否已变化。

**Why:** 用户当前的核心痛点是“我看不明白现在到底结束没有”，但用户也已经明确表示状态码本身足够表达完成情况。去掉额外状态标签后，视觉注意力可以留给真正有价值的异常信息。

**Alternative considered:** 继续给状态码叠加一层“已完成/已失败/已取消/已中断”文字标签。  
**Rejected because:** 额外标签会增加视觉噪声，反而稀释真正重要的异常信息。

## 可视化设计稿

### 1) 实时刷新与状态收口总流程

```text
┌────────────────────┐
│ user sends request │
└─────────┬──────────┘
          ▼
   logRequestStart()
          ▼
   request_logs row
   status = 进行中
          │
          ├──────── success / error ────────┐
          │                                  ▼
          ├──────── client cancel ───────► terminal settle
          │                                  │
          ├──────── stream interrupted ───► terminal settle
          │                                  │
          └──────── no settle for too long ─► stale reconcile
                                             ▼
                                   logs snapshot + live hint
                                             ▼
                                      user sees final state
```

### 2) 日志页关键场景布局示意

```text
┌────────────────────────────────────────────────────────────────────┐
│ 请求日志                                                          │
│ [实时更新中]                              [手动刷新] [筛选条件...] │
├────────────────────────────────────────────────────────────────────┤
│ 200      gpt-5.2   OpenAI   2s ago                                  │
│ 499      claude-*  Anthropic  刚刚                                  │
│ 499      codex-*   OpenAI    3s ago                                 │
│ [spinner] gemini-* Gemini   实时变化中                              │
└────────────────────────────────────────────────────────────────────┘
```

### 3) 视觉层级说明

| 层级 | 内容 | 目的 |
|---|---|---|
| L1 | 状态码或进行中加载指示 | 用户第一眼确认请求是否已经结束 |
| L2 | 关键模型/上游信息 + 红色错误信息 | 便于快速定位问题范围 |
| L3 | 实时变化提示或更新时间 | 让用户感知“系统还在更新” |
| L4 | 展开详情中的失败阶段、证据和原始摘要 | 用于排障，不干扰主列表判断 |

## Risks / Trade-offs

- [风险] 实时刷新链路断开后用户误以为页面仍在自动更新 -> [缓解] 明确展示实时状态指示，并在链路不可用时自动回退到定时刷新。
- [风险] 499 属于约定俗成状态码，部分历史日志没有同等语义 -> [缓解] 保留 failure stage 与错误详情辅助解释，并用错误色突出异常信息。
- [风险] stale reconciliation 阈值过短会误杀慢请求 -> [缓解] 使用保守默认值，并要求只收敛明显超过合理生命周期上限的请求。
- [风险] 在日志查询前做 best-effort reconcile 会增加少量读路径开销 -> [缓解] 仅针对进行中小集合执行，并限制批次大小。

## Migration Plan

1. 先修正代理路径中的终态收口，确保取消/中断不会长期停留在进行中。
2. 在类型转换和 API 层补齐用户可见状态映射与兼容字段输出。
3. 引入轻量实时刷新链路，并在前端添加自动更新与回退逻辑。
4. 在日志查询和统计聚合前加入 stale reconciliation，收敛历史脏状态。
5. 发布后重点观察：进行中数量、499/中断终态占比、日志页刷新体验反馈。

## Open Questions

- stale reconciliation 的默认阈值应该以“分钟”为单位配置，还是直接复用上游 timeout 的倍数策略？
- 用户可见文案更适合统一为“已取消”，还是区分“调用方已取消”和“传输已中断”？
