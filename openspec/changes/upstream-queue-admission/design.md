## Context

当前主干已经具备三项与本次变更强相关的基础能力：

1. `load-balancer.ts` 使用单实例内存 `connectionCounts` 管理上游活跃连接数，并在选择阶段通过原子保留槽位避免并发竞争。
2. 代理入口在所有候选上游都因并发被排除时，会返回 `CONCURRENCY_FULL`，同时保留“容量满载不触发熔断”的语义。
3. 管理端已经能够展示 `max_concurrency`、`current_concurrency`、`concurrency_full` 对应的运行态与日志证据。

当前缺失的是“并发满载后的等待准入”这一整层能力。也就是说，系统知道何时满载，也知道如何转移或失败，但还没有“等待一个即将释放的槽位”这一行为本身。

本次变更的直接干系人分为两类：

- 管理员：需要为不同上游配置不同的等待策略，并在管理端看见与排队相关的配置和结果。
- 排障人员：需要在日志中区分“请求是直接失败”“请求等待后恢复执行”“请求等待超时”“请求等待期间客户端断开”。

本次变更同时跨越数据模型、运行时准入、统一错误语义、日志与前端展示，属于典型的跨模块设计改动。

**关键运行时路径**

```text

候选筛选
   |
   +--> 有立即可用上游 ----------> 保留槽位 ----------> 发往上游
   |
   +--> 全部因并发被排除 ---------> 直接返回 CONCURRENCY_FULL


目标路径

候选筛选
   |
   +--> 有立即可用上游 ----------> 保留槽位 ----------> 发往上游
   |
   +--> 全部因并发被排除
            |
            +--> 目标上游未启用排队 ----------> 返回 CONCURRENCY_FULL
            |
            +--> 目标上游启用排队
                     |
                     +--> 队列可接纳 ----------> waiting
                     |                           |

                     |                           |
                     |                           +--> 超时 ----> queue timeout
                     |                           |
                     |                           +--> 客户端断开 -> aborted
                     |
                     +--> 队列已达上限 ----------> 返回容量相关错误
```

## Goals / Non-Goals

**Goals:**

- 为上游提供结构化的排队策略配置，并保持与现有 `max_concurrency` 语义一致。
- 在所有即时可用候选都满载时，支持单实例内存 FIFO 等待队列，而不是立刻失败。
- 将等待超时与上游请求超时明确区分，并保留独立的错误语义、用户提示和日志字段。
- 在请求结束、流式关闭、异常失败和客户端断开时，确保槽位释放和队首唤醒顺序一致。
- 在管理端日志中完整展示排队生命周期，包括等待、恢复执行、超时和中断。

**Non-Goals:**

- 本次不实现跨实例共享队列、分布式信号量或 Redis 协调。
- 本次不引入全局队列；排队单位保持为上游级队列。
- 本次不改变现有 tier、quota、session affinity 的优先级规则，只在“所有即时候选都满载”后扩展等待行为。
- 本次不重做日志页与上游页整体视觉体系，只在现有结构上补充队列相关状态与配置。

## Decisions

### 决策 1：新增独立的队列准入服务，统一管理活跃槽位与等待队列

**选择**

- 新增 `upstream-queue-admission` 运行时服务，按上游维护单实例内存状态。
- 该服务持有两类原子状态：活跃连接数、FIFO 等待队列。
- 现有 `connectionCounts` 迁移为该服务内部实现细节，对外暴露“尝试立即保留”“进入等待”“释放并唤醒”接口。

**理由**

- 如果仍把 `connectionCounts` 和等待队列分散在不同模块，槽位释放与队首唤醒之间会出现竞争窗口，新请求可能抢在队首之前拿到槽位，破坏 FIFO 语义。
- 把“计数 + 队列 + 唤醒”合并到同一服务，可以将所有并发相关状态限制在一个内存边界内，便于测试与推理。

**替代方案**

- 在 `load-balancer.ts` 里继续堆叠等待逻辑：可以实现，但状态边界会与选择逻辑耦合过深。
- 让 `proxy route` 自己维护等待 Promise：调用层会知道太多运行时细节，难以保证释放顺序一致。

### 决策 2：只有在所有即时候选都满载后，才允许进入绑定上游的 FIFO 队列

**选择**

- 保持现有即时选择策略不变：同 tier 有可用上游时继续转移；当前 tier 满载但下一 tier 可用时继续降级。
- 只有在所有符合当前规则的候选上游都因并发满载而无法立即选择时，才进入等待逻辑。
- 队列条目绑定到最终决定等待的那个上游，而不是全局共享候选集。

**理由**

- 这与 issue 本身的约束一致：等待是“即时可用候选全部耗尽”后的补充路径，不应抢先于已有的即时转移与降级机制。
- 上游级队列更容易解释容量语义，也更符合不同提供商响应时间差异较大的现实。

**替代方案**

- 只要首选上游满载就立即排队：会削弱现有 failover 价值，导致本来可立即成功的请求被动等待。
- 采用全局队列：会混合不同上游的容量与优先级，行为难以解释。

### 决策 3：排队策略使用结构化字段，而不是继续复用原始 `config` 文本

**选择**

- 为上游新增结构化 queue policy，至少包含：
  - `enabled`
  - `timeout_ms`
  - `max_queue_length`
- PostgreSQL 与 SQLite schema 同步新增字段，API 与前端表单以显式字段对外暴露。

**理由**

- 当前仓库对熔断器、会话亲和性迁移、消费限额都采用结构化字段；排队策略继续沿用这一风格，能减少解析和合并错误。
- 原始 `config` 文本列已经不是主流配置载体，再把新策略塞回去，只会让变更边界变得模糊。

**替代方案**

- 将 queue policy 存进 `config` 文本：迁移成本较低，但局部更新、类型校验和脏值比较都更脆弱。

### 决策 4：等待超时使用独立网关语义，与上游请求超时彻底分离

**选择**

- 新增独立的统一错误代码与原因，用于表达“请求已进入等待队列但在获得执行资格前超时”。
- 等待超时沿用网关侧错误处理路径，不计入上游失败，不写入熔断器失败计数。
- `durationMs` 继续记录请求总耗时；等待耗时单独进入 `routingDecision.queue.wait_duration_ms`。

**理由**

- 等待超时发生在请求尚未发往上游之前，本质上不是网络超时，也不是上游超时。
- 如果复用 `REQUEST_TIMEOUT`，调用方与排障人员都会误判问题发生位置。

**替代方案**

- 复用现有 `REQUEST_TIMEOUT`：实现简单，但会混淆“等槽位超时”和“上游响应超时”。

### 决策 5：队列观测继续挂在 `routingDecision` 下，不新建第二套日志结构

**选择**

- 在 `RoutingDecisionLog` 下新增 `queue` 子对象，记录：
  - `status`
  - `upstream_id`
  - `entered_at`
  - `resumed_at`
  - `wait_duration_ms`
  - `timeout_ms`
  - `is_stream`
- 前端日志列表和时间线从 `routingDecision.queue` 读取队列状态。

**理由**

- 当前日志体系已经把选择、排除、最终上游、失败阶段都放在 `routingDecision`，队列只是路由决策的延展状态。
- 继续使用单一承载结构，能减少 API 扩散和前后端 DTO 分裂。

**替代方案**

- 新建顶层 `queue` 日志字段：结构更分散，需要额外同步逻辑。

### 决策 6：槽位释放必须采用“保留后移交”的唤醒策略

**选择**

- 当执行中的请求结束时，先在准入服务内部把释放出的槽位移交给队首等待项，再唤醒该等待项继续执行。
- 队首恢复后才真正进入现有 `forwardRequest` 路径。
- 如果等待项在被唤醒前已超时或客户端已断开，则跳过该项并继续检查下一项。

**理由**

- 单纯先减计数再异步通知，会让刚到达的新请求抢在队首之前，破坏 FIFO。
- “保留后移交”可以把等待恢复与即时新请求统一在同一原子边界中。

**替代方案**

- 普通 `releaseConnection()` 后再 `resolve()` 队首 Promise：实现直观，但存在竞争窗口。

### 决策 7：被唤醒的等待请求在真正发往上游前做一次轻量复核

**选择**

- 队列条目恢复执行后，对绑定上游做一次轻量复核，确认其仍处于启用状态，并允许当前请求进入现有代理路径。
- 如果绑定上游在等待期间已经被停用、删除或变成明显不可用，则释放保留槽位，并回到候选选择流程做一次重新选择。

**理由**

- 等待期间系统状态可能发生变化。完全信任旧快照会让请求在恢复执行时撞到过时配置。
- 只做一次轻量复核和单次重选，可以控制复杂度，避免无限回流。

**替代方案**

- 完全不复核：实现最简单，但边界条件脆弱。
- 每次恢复都完全重新做整套候选决策：行为更稳，但会让排队绑定语义变得模糊。

## Risks / Trade-offs

- [单实例队列与多实例部署语义不一致] → 明确将本次能力限定为单实例内存队列，并在文档与规格中写清边界。
- [槽位释放与唤醒顺序实现错误会破坏 FIFO] → 将计数与队列收口到单一准入服务，并为“保留后移交”补充单元测试。
- [等待时间混入既有性能指标会污染统计] → 总耗时与等待耗时分离记录，等待细节只进入 `routingDecision.queue`。
- [客户端在等待期断开会遗留脏队列项] → 队列项必须携带 abort signal，并在超时、断开和恢复前都做移除清理。
- [队列配置过大可能形成无边界内存占用] → 支持可选 `max_queue_length`，并在管理端提供明确配置入口。
- [复核与重选会让恢复路径复杂化] → 只允许一次轻量复核与单次重选，避免把等待恢复变成第二套完整路由器。

## Migration Plan

1. 在 PostgreSQL 与 SQLite schema 中新增 queue policy 结构字段，并为 service/API/types 打通读写链路。
2. 引入新的准入服务，先让现有 `connectionCounts` 读取迁移到该服务，再加入 FIFO 队列与保留移交逻辑。
3. 在代理入口接入“全部候选满载后等待”的分支，并增加独立的等待超时错误语义。
4. 扩展 `RoutingDecisionLog` 与请求日志转换层，记录队列状态和等待耗时。
5. 在上游配置弹窗与日志界面补充队列配置和队列观测展示。
6. 以单元测试和组件测试覆盖“即时成功、等待恢复、等待超时、客户端断开、流式释放槽位”五类关键路径。
7. 回滚时可先关闭 queue policy 配置，使系统恢复到现有“满载即转移或失败”的行为，而无需先删除 schema 字段。

## Open Questions

- 当 `max_queue_length` 已满而请求尚未进入等待队列时，是否需要独立的下游错误原因，还是继续沿用容量相关错误语义。
- 被唤醒请求在轻量复核时，如果发现绑定上游已被熔断打开，单次重选是否只限当前 capability 候选集，还是保留与即时路径完全一致的 tier 降级行为。
- 管理端上游列表是否需要展示实时排队长度；本次 issue 只要求配置和日志，实时排队长度可以后续再评估。
