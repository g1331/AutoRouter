## Context

当前仓库已经以路径能力为主进行上游选择。请求先命中 `RouteCapability`，再经过授权、可用性和故障转移处理。上游管理端则以 `src/components/admin/upstream-form-dialog.tsx` 为统一编辑入口，当前“模型路由”区只提供 `allowed_models` 标签输入和 `model_redirects` 键值输入。

这带来三类现实约束：

1. 运行态已经以路径能力为核心，新的模型规则必须插入现有候选过滤顺序，不能引入另一套 `provider_type` 驱动结构。
2. 上游测试与兼容接口目前会把 `base_url` 规整到 `origin`，这会丢失 `/codex/v1` 之类的 API 根路径前缀，模型发现如果沿用同样处理方式会继续出现地址错误。
3. issue 110 需要同时覆盖“模型目录发现”“模型规则表达”“管理端导入”三个方面，涉及数据模型、代理入口、管理端界面和测试，因此需要在编码前先固定技术决策。

当前状态示意如下：

```text
当前 master

Request
  │
  ├─ 路径能力命中
  ├─ API Key 授权
  ├─ 可用性选择
  └─ 转发

Upstream Form
  │
  └─ allowed_models + model_redirects
```

本次变更后的目标结构如下：

```text
Request
  │
  ├─ 路径能力命中
  ├─ API Key 授权
  ├─ 模型规则匹配
  ├─ 可用性选择
  └─ 转发

Upstream Form
  │
  ├─ 模型规则编辑
  ├─ 目录刷新状态
  └─ 目录浏览与导入
```

## Goals / Non-Goals

**Goals:**

- 为每个上游引入统一的模型发现配置、目录缓存和最近刷新状态。
- 为每个上游引入统一的模型规则结构，覆盖精确模型名、正则模式和人工别名。
- 在路径能力选路后增加模型规则过滤阶段，使显式规则真正成为约束条件。
- 在管理端上游编辑界面增加目录刷新、目录浏览、来源标记和导入动作。
- 保持当前路径能力、授权、优先级、权重、并发容量、熔断和故障转移算法可复用。
- 为现有 `allowed_models` 与 `model_redirects` 历史数据提供兼容迁移语义。

**Non-Goals:**

- 不恢复 `provider_type` 字段，也不把代理入口改成“模型前缀决定上游分组”的另一套结构。
- 不在本次变更中引入全局定时刷新服务；目录刷新以管理员触发为主。
- 不自动删除管理员手工添加的规则；原生发现刷新只更新目录缓存，不直接覆写人工规则。
- 不改写现有负载均衡、并发容量、熔断和故障转移算法本身，只调整它们之前的候选过滤顺序。

## Decisions

### Decision 1：引入规范化的模型发现与模型规则字段，并保留历史字段兼容语义

上游数据模型新增以下规范化字段：

```text
model_discovery
  mode: openai_compatible | anthropic_native | gemini_native | gemini_openai_compatible | custom | litellm
  custom_endpoint: string | null
  enable_lite_llm_fallback: boolean

model_catalog: [{ model, source }]
model_catalog_updated_at
model_catalog_last_status
model_catalog_last_error
model_catalog_last_failed_at

model_rules:
  - type: exact
    display_label: 精确匹配
    meaning: 请求模型名与规则值完全一致时命中
  - type: regex
    display_label: 模式匹配
    meaning: 请求模型名满足正则表达式时命中
  - type: alias
    display_label: 模型别名
    meaning: 请求模型名命中后改写为目标模型名再转发
```

其中 `model_rules` 作为新的规范化规则集合，`allowed_models` 与 `model_redirects` 视为历史兼容输入与输出：

- 旧行数据读取时，服务层会把 `allowed_models` 和 `model_redirects` 规范化为规则视图。
- 新写入路径以 `model_rules` 为准，并在迁移窗口内继续输出历史字段，避免现有调用方立即失效。
- 手工规则与目录导入规则共存，规则条目保留 `source=manual | native | inferred` 以标记来源。
- `type` 是内部规则类型值，管理端界面展示使用 `display_label` 对应的中文名称，避免把实现枚举直接暴露为管理员文案。

选择这条方案的原因：

- 可以在当前仓库上平滑升级，而无需一次性移除历史字段。
- 运行时和管理端都可以只围绕一套规则语义工作，避免双套逻辑长期并存。

备选方案：

- 立即废弃 `allowed_models` 与 `model_redirects`。这会缩短数据模型，但升级窗口过陡，现有 API 和测试需要同轮全部改写。

### Decision 2：模型发现使用“已配置 API 根路径”而不是单纯 `origin`

模型发现与连接测试都以“归一化后的 API 根路径”为基准，而不是直接退回 `origin`。归一化规则是：

- 保留 `base_url` 中已经存在的路径前缀，例如 `/codex/v1`。
- 只清理多余尾部斜杠和重复路径片段。
- 每种发现模式在“API 根路径”上追加自己的目录路径。

发现模式与目标地址的关系如下：

| 模式 | 发现地址语义 | 认证方式 |
| --- | --- | --- |
| `openai_compatible` | `<api-root>/models` | `Authorization: Bearer` |
| `anthropic_native` | `<api-root>/models` | `x-api-key` + `anthropic-version` |
| `gemini_native` | `<api-root>/models?key=...` | query key |
| `gemini_openai_compatible` | `<api-root>/models` | `Authorization: Bearer` |
| `custom` | 自定义 endpoint，相对路径基于 `<api-root>` 解析 | 跟随能力族认证 |
| `litellm` | LiteLLM 公共目录 | 无上游私钥参与 |

选择这条方案的原因：

- 与 issue 110 需要支持的兼容网关场景一致。
- 可以复用现有 endpoint 预览语义，降低管理员理解成本。

备选方案：

- 始终退回 `origin` 再拼接固定路径。这个方案实现简单，但会继续破坏带子路径的兼容接口。

### Decision 3：代理入口在授权之后插入模型规则过滤，严格语义按“上游维度”生效

当前路径能力主链保持不变，只在授权候选集之后插入模型规则匹配：

```text
请求进入
  │
  ├─ 路径能力命中
  ├─ API Key 授权
  ├─ 逐个上游执行模型规则匹配
  │    ├─ 无显式规则：继续保留
  │    ├─ 命中精确匹配 / 模式匹配：保留
  │    ├─ 命中模型别名：保留，并解析目标模型
  │    └─ 存在显式规则但未命中：排除
  ├─ 可用性选择
  └─ 转发
```

严格语义按“上游维度”生效，而不是“全局维度”生效：

- 某个上游配置了显式规则且本次请求未命中，该上游从候选集中排除。
- 某个上游没有显式规则，仍可以视为开放候选。
- 只有别名规则会改变最终转发模型名；正则规则只负责“允许匹配”，不隐式改写模型名。

选择这条方案的原因：

- 这与 issue 110 的“显式配置了白名单规则就必须尊重”完全一致。
- 对当前路径能力架构改动最小，现有可用性选择与故障转移代码可复用。

备选方案：

- 所有上游只要出现任一显式规则，就让整个能力族进入“全局严格模式”。这种做法会让未配置规则的上游也被连带收紧，超出 issue 110 的要求。

### Decision 4：管理端将模型路由区设计为单主画布工作区，桌面双栏，移动端单栏

`upstream-form-dialog` 保持现有长表单和目录导航结构，但“模型路由”分区本身采用单主画布工作区，而不是把规则、状态和目录切成多个同权重卡片。该工作区包含三层信息：

1. 顶部紧凑状态条，承载发现模式、刷新动作、最近状态和选择计数。
2. 主工作区，桌面端采用“规则编辑为主，目录浏览为辅”的双栏布局。
3. 次级反馈区，承载空态、加载态、错误态和导入反馈。

桌面端布局示意：

```text
+--------------------------------------------------------------------------------------------------+
| 模型路由                                                                                          |
| [发现模式] [刷新目录] [最近成功/失败] [最后刷新时间] [已选 3 项]                                  |
|--------------------------------------------------------------------------------------------------|
| 规则编辑 5/12                                        | 目录浏览 7/12                              |
|------------------------------------------------------|-------------------------------------------|
| 精确匹配                                              | 搜索 / 来源筛选 / 来源说明                 |
| 模式匹配                                              |-------------------------------------------|
| 模型别名                                              | source   model name              select    |
|------------------------------------------------------| native   gpt-4.1                [x]       |
| 规则命中提示 / 别名预期                              | inferred gpt-4.1-preview        [ ]       |
|                                                      | native   gpt-4.1-mini           [x]       |
|------------------------------------------------------|-------------------------------------------|
| 导入预览摘要                                         | 空态 / 加载态 / 错误态在表格区就地呈现     |
| [导入所选模型]                                       |                                           |
+--------------------------------------------------------------------------------------------------+
```

移动端布局示意：

```text
+-----------------------------------------------------------+
| 模型路由                                                   |
| [发现模式] [刷新目录]                                      |
| [最近状态] [最后刷新时间] [已选 3 项]                      |
|-----------------------------------------------------------|
| 规则编辑                                                   |
|  - 精确匹配                                                |
|  - 模式匹配                                                |
|  - 模型别名                                                |
|-----------------------------------------------------------|
| 目录浏览                                                   |
|  搜索 / 来源筛选                                           |
|  native   gpt-4.1                            [x]           |
|  inferred gpt-4.1-preview                    [ ]           |
|-----------------------------------------------------------|
| 导入摘要                                                   |
| [导入所选模型]                                            |
+-----------------------------------------------------------+
```

视觉层级说明：

- 第一视觉焦点是规则编辑器，因为它定义运行态约束。
- 第二视觉焦点是顶部状态条，因为它回答“目录是否可信、是否需要刷新”。
- 第三视觉焦点是目录浏览区，因为它属于批量辅助配置。
- 界面文案采用运营工具语气，只展示状态、范围和动作，不引入宣传式描述。
- 在现有弹窗内部避免再叠加一组卡片式小容器，优先使用分区、分隔线、表头和紧凑状态条维持扫描节奏。

选择这条方案的原因：

- 规则编辑和目录浏览在同一工作面内，关系清晰，但视觉权重不会平均分散。
- 桌面端可以提升扫描效率，移动端继续保持线性阅读顺序。
- 与现有长表单导航兼容，不需要把目录浏览拆成独立页面。

备选方案：

- 全部采用单列上下堆叠。这个方案实现简单，但桌面端的扫描效率和批量导入效率都偏弱。
- 把目录浏览拆到独立页面。这个方案可以降低弹窗密度，但会割裂“编辑规则”和“导入目录”的连续操作。

### Decision 5：LiteLLM 只作为目录候选补充，不作为真实支持证明

当 `enable_lite_llm_fallback=true` 且原生发现失败时，系统会读取 LiteLLM 公共目录，并把条目标记为 `source=inferred`：

- 这些条目可以展示和导入。
- 这些条目不能被描述为“上游真实返回模型”。
- 刷新结果需要明确记录本次是否使用了 fallback。

选择这条方案的原因：

- LiteLLM 对别名与模型覆盖较广，适合作为目录建议来源。
- 保持来源区分后，管理员仍然能判断哪些条目需要进一步核实。

备选方案：

- 原生发现失败即完全终止。这个方案更保守，但会使私有兼容接口和代理转发场景缺少初始配置辅助。

## Risks / Trade-offs

- [规则模型与历史字段并存，语义可能出现偏差] → 统一由服务层做规范化读写，禁止前端同时独立维护两套规则状态。
- [正则规则过宽，可能放入管理员原本未预期的模型名] → 在管理端展示示例匹配说明，并在测试中覆盖高风险模式。
- [LiteLLM 目录与真实上游能力不一致] → 明确标记 `inferred` 来源，并把 fallback 使用情况写入最近刷新状态。
- [目录缓存字段增加后，本地 SQLite 容易出现 schema 漂移] → PostgreSQL 与 SQLite 同步补迁移，验证脚本中加入一致性检查。
- [目录刷新、导入、规则编辑都放进同一弹窗后，内容密度上升] → 采用单主画布、紧凑状态条、桌面双栏和移动端单栏，避免出现同权重卡片拼盘。

## Migration Plan

1. 为 PostgreSQL 与 SQLite 的 `upstreams` 表新增模型发现、目录缓存和规则字段，并补齐迁移脚本。
2. 在 API transformer、类型定义和 CRUD 服务中加入新字段的读写与兼容转换。
3. 新增模型发现服务与模型规则服务，并先用单元测试固定“地址解析”“规则匹配”“历史字段兼容”语义。
4. 在代理入口接入“授权之后、可用性之前”的模型规则过滤。
5. 在管理端上游编辑界面加入状态条、规则编辑区、目录浏览区和导入反馈区，并按桌面双栏、移动端单栏实现。
6. 为历史数据提供读取时推断和空值兼容，目录缓存初始为空，等待管理员首次刷新。
7. 完成回归测试后发布。

回滚策略：

- 回退代码到旧版本，并停止读取新增字段。
- 数据库新增列可以保留为惰性字段，不要求同步删除。
- 如果回滚发生在规则导入之后，旧版本继续使用 `allowed_models` 与 `model_redirects` 兼容视图，不读取规范化规则字段。
