## Context

现有代理路径中，`forwardRequest` 只在 `fetch()` 等待响应头阶段使用上游基础 `timeout`。流式响应返回后，代理会将上游 body 包装后直接交给下游，并通过 `createSSETransformer` 提取 usage 和 TTFT。现有 TTFT 判定已经能排除纯元数据事件，因此可复用为“首字已经到达”的信号。

熔断计数由代理路由在失败或非 2xx 响应后调用 `recordFailure` 完成。现有 `FailoverConfig.excludeStatusCodes` 只能静态排除状态码，不支持按响应体、错误类型、全局规则或上游局部规则决定是否计入熔断。

```
下游请求
  |
  v
候选选择 -> 上游请求 -> 响应头
                         |
          +--------------+--------------+
          |                             |
      非流式响应                    SSE 流式响应
          |                             |
   状态码/响应体判定            首字等待 -> 空闲检测 -> 下游输出
          |                             |
          +--------------+--------------+
                         |
              故障转移记录 + 熔断计数判定
```

## Goals / Non-Goals

**Goals:**
- 支持首字超时与流式空闲超时，并与现有故障转移、健康状态和熔断计数结合。
- 支持可管理的失败规则，让命中规则的失败继续重试但不计入熔断。
- 在日志和管理端中保留失败证据、规则命中信息和熔断计数影响。
- 保持流式实时输出；已向下游发送内容后发生空闲超时，不尝试拼接其他上游响应。

**Non-Goals:**
- 不引入跨实例共享的失败规则缓存或分布式队列。
- 不对已发送给下游的流式内容做回滚、重放或跨上游拼接。
- 不改变 API key 鉴权、模型规则、配额或计费语义。

## Decisions

### 失败规则模型

新增 `upstream_failure_rules` 表，使用 `upstream_id = null` 表示全局规则，非空表示上游本地规则。规则包含启用状态、名称、匹配条件和创建更新时间。匹配条件使用 JSON 保存，第一版支持：

| 条件 | 含义 |
| --- | --- |
| `status_codes` | 匹配一个或多个上游 HTTP 状态码 |
| `error_types` | 匹配代理内部失败类型 |
| `body_pattern` | 使用正则匹配响应体或错误消息 |
| `header_name` / `header_pattern` | 使用正则匹配指定响应头 |

上游是否使用全局规则保存在 `upstreams.failure_rule_config.use_global_rules` 中，默认 `true`。读取规则时合并全局规则和本地规则，本地规则始终参与匹配。

### 超时配置位置

首字超时和空闲超时加入 `circuit_breaker_states.config`，字段为 `firstByteTimeout` 与 `streamIdleTimeout`，内部单位毫秒，API 与管理端单位秒。默认值放在 `DEFAULT_CIRCUIT_BREAKER_CONFIG`，避免已有上游必须补写配置。

### 首字与空闲判定

首字判定复用现有 `createSSETransformer` 的有效输出事件识别。首字计时从 `fetch()` 发出开始；响应头到达但一直只有元数据事件时，仍然可能触发首字超时。

空闲计时按上游原始 body chunk 刷新。只要上游有任何 SSE 字节到达，就视为连接仍有进展；如果超过配置时间没有新 chunk，则抛出 `StreamIdleTimeoutError`。

### 故障转移与下游输出边界

```
首字前超时
  上游尚无有效内容 -> 可换上游 -> 计入失败规则/熔断判定

首字后空闲超时
  下游可能已收到内容 -> 结束当前流 -> 记录失败 -> 不换上游
```

首字前失败发生在返回下游 `Response` 前，可以继续无感尝试其他上游。首字后失败发生在 downstream streaming 阶段，只能以当前请求终态记录。

### 管理端布局

```
上游编辑弹窗
┌────────────────────────────────────────────┐
│ 高级设置                                   │
│ ┌ 熔断器 ────────────────────────────────┐ │
│ │ 失败阈值  成功阈值                     │ │
│ │ 打开时长  探测间隔                     │ │
│ │ 首字超时  响应空闲超时                 │ │
│ └────────────────────────────────────────┘ │
│ ┌ 失败规则 ──────────────────────────────┐ │
│ │ [开关] 使用全局失败规则                │ │
│ │ 本地规则列表：名称 / 状态码 / 错误类型 │ │
│ │ 响应体正则 / 响应头匹配 / 启用状态     │ │
│ └────────────────────────────────────────┘ │
└────────────────────────────────────────────┘
```

失败规则配置放在高级设置中，视觉层级低于基础路由和配额配置；列表采用紧凑表单行，避免占用主要配置区域。

全局失败规则管理放在系统设置页下的独立页面。系统设置页提供入口，页面复用本地失败规则编辑能力，但数据来源改为全局失败规则 API。上游编辑弹窗继续保留“使用全局失败规则”开关和本地规则编辑，避免把全局维护操作混入单个上游编辑流程。

## Risks / Trade-offs

- 正则匹配可能写得过宽 → 管理端展示规则名称，并在日志中记录命中规则，便于回查。
- 已发送流式内容后无法无感重试 → 保持实时输出优先，日志明确标识 `stream_idle_timeout`。
- 规则表新增迁移会影响 PostgreSQL 与 SQLite → 同步更新双 schema、迁移和一致性检查。
- 响应体匹配需要读取失败响应体 → 只在非 2xx 或异常证据捕获路径读取，并沿用现有错误体截断与脱敏边界。
