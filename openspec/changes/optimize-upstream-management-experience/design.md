## Context

当前上游管理能力分散在多个位置：上游配置在 `upstream-form-dialog`，运行态信息在 `upstreams-table`，路由证据在 `logs-table` 与 `routing-decision-timeline`。  
现状存在三个核心问题：

1. Endpoint 规则依赖用户记忆（例如 base URL 是否包含 `/v1`），缺乏配置期反馈与运行期可解释性。
2. 系统已有连接计数器与失败转移机制，但并发容量没有显式配置与可视化，用户无法提前控制“满载时如何转移”。
3. 上游页以高密度表格承载过多职责（配置、状态、配额、操作），在扩展官网地址、并发占用、last used 后会进一步失衡。

约束：

- 必须保持“并发满载不触发熔断”的语义，避免把容量限制误判为健康故障。
- 必须兼容现有 failover 机制、quota 机制与 session affinity 机制。
- 必须同时支持桌面与移动端一致的信息语义。

关键干系人：

- 管理员：需要在上游页快速识别容量、健康、配额和最近活跃状态，并快速跳转官网/编辑。
- 运行与排障人员：需要在日志中准确识别“因为并发满而转移”的链路证据。

## Goals / Non-Goals

**Goals:**

- 提供“低认知负担”的 endpoint 配置体验：自动补全、重复警告、最终请求地址预览。
- 为每个上游提供并发上限控制，并在运行态与日志中完整可见。
- 将上游页重构为运营工作台式信息架构，提升视觉整齐度与扩展性。
- 将 `created_at` 替换为 `last_used_at`，让列表信息更贴近运维决策。

**Non-Goals:**

- 本次不引入跨实例分布式并发令牌（如 Redis 全局信号量）。
- 本次不实现请求排队与队列调度，仅做“满载转移”。
- 本次不重做整体设计系统，仅在现有视觉语言内完成上游页重构。

## Decisions

### 决策 1：Endpoint 采用“能力感知自动补全 + 预览”而不是全局硬编码

**选择**：

- 在上游配置阶段，根据能力集合决定 base URL 归一化策略。
- 对 OpenAI/Anthropic/Codex 相关能力支持自动补全 `/v1`。
- 用户手动输入末尾 `/v1` 且会导致重复时显示警告，不阻塞保存。
- 使用“最终请求地址预览框”展示实际拼接结果。

**理由**：

- 满足“用户填写 `https://.../codex/` 即可”的目标，同时保持可解释性。
- 避免对 Gemini `v1beta` / `v1internal` 等路径产生误补。

**替代方案**：

- 全局强制补 `/v1`：实现简单但会误伤非 `/v1` 能力。
- 完全不补：继续把规则负担留给用户，不满足体验目标。

### 决策 2：并发控制采用“每上游硬上限 + 满载排除”并保持无熔断语义

**选择**：

- 新增 `max_concurrency`（可空，空表示不限流）。
- 路由候选筛选新增并发过滤层：并发达到上限的上游从当前请求候选中排除。
- 排除原因记为 `concurrency_full`，触发同 tier 重选或跨 tier 降级。
- 并发满载路径不调用 `markUnhealthy` 与 `recordFailure`。

**理由**：

- 容量限制属于“负载状态”，不是“服务异常”，不应污染熔断状态机。
- 和现有连接计数生命周期一致，改造面相对可控。

**替代方案**：

- 软限制（仅提示不拦截）：无法保证容量约束。
- 请求排队：行为复杂且改变延迟模型，不适合本次迭代。

### 决策 3：日志与路由决策增加并发满证据，形成“配置-运行-排障”闭环

**选择**：

- 扩展 failover attempt 错误类型：增加 `concurrency_full`。
- 扩展 routing excluded reason：增加 `concurrency_full`。
- 路由时间线 Stage 3/4 显示“并发满导致排除与转移”。
- 在日志详情中展示转移前后上游与并发状态摘要。

**理由**：

- 用户可直接回答“为什么没走预期上游”，减少黑盒感。
- 与现有 failover history / routing decision 模型兼容。

**替代方案**：

- 仅在服务端日志记录，不上屏：运营人员仍需翻后端日志，效率低。

### 决策 4：上游页改为“Tier + 卡片工作台”，不再以表格作为主承载

**选择**：

- 页面骨架仍保留顶部管理区，但列表主视图改为分组卡片布局。
- 桌面与移动共享同一信息块语义，差异仅在排版密度。
- 操作按钮分层：主操作直显，次操作收敛，减少噪声。

**布局示意（桌面）**：

```text
┌──────────────────────────────────────────────────────────────────────┐
│ Upstream Operations Bar                                              │
│ [Search] [Capability Filter] [Status Filter] [Add Upstream]          │
└──────────────────────────────────────────────────────────────────────┘

┌──────────────── Tier P0 (Healthy 3/4 | Full 1 | Exceeded 0) ────────┐
│ ┌──────────────────── Upstream Card ────────────────────────────────┐ │
│ │ Name + Capability Badges + Active Switch                          │ │
│ │ Endpoint: Base URL / Final URL Preview / Official Site            │ │
│ │ Runtime : Health | Circuit | Quota | Concurrency (3/10)           │ │
│ │ Last Used: 2 min ago                                               │ │
│ │ Actions: [Test] [Edit] [More]                                     │ │
│ └────────────────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────────────┘
```

**布局示意（移动）**：

```text
[Tier Header]
  └─ [Card]
     Name + Status
     Endpoint + Preview
     Runtime chips
     Last Used
     Action row
```

**视觉层级说明**：

| 层级 | 信息类型 | 视觉权重 |
|---|---|---|
| L1 | 上游名称、关键状态（满载/熔断/超额） | 高对比 + 明确状态色 |
| L2 | Endpoint 与并发占用、配额进度 | 中对比 + 结构化块 |
| L3 | 官网链接、次级元数据（last used） | 低对比 + 辅助文案 |

### 决策 5：`last_used_at` 取代列表 `created_at`

**选择**：

- 列表项显示最近一次使用时间，空值显示“未使用”。
- 数据通过请求日志按 upstream 聚合 `MAX(created_at)` 获取。

**理由**：

- 创建时间对日常运维决策价值低。
- 最近使用时间可直接反映上游活跃度与冗余配置。

**替代方案**：

- 同时显示两者：信息噪声上升，优先级不清晰。

### 决策 6：上游配置弹窗提供“快速定位导航”并支持一键跳转

**选择**：

- 在 `upstream-form-dialog` 的高级配置区域提供可见的配置导航入口。
- 用户点击导航项后，系统直接滚动到对应配置区块（高级配置当前为常显，无需展开；例如计费倍率、消费限额、模型路由、熔断器、会话迁移）。
- 配置导航在高级配置展开后保持可达（粘性导航），并在各区块提供“上一项/下一项/返回快速定位”闭环操作。
- 桌面端提供左侧目录式导航（分组 + 图标 + 搜索），移动端降级为顶部紧凑导航，保持同一跳转语义。
- 弹窗标题区与目录区固定，表单内容在中间区域独立滚动，避免滚动后丢失定位。
- 保存/取消操作栏固定在底部，确保长表单编辑时无需滚动到底部即可提交。
- 当表单存在未保存改动且用户尝试关闭弹窗（含点击遮罩或取消）时，弹出确认提示避免误丢失。
- 导航跳转后对目标区块进行短时高亮，明确当前编辑焦点。
- 导航文案与区块标题统一走 i18n，保证中英文一致可理解。

**理由**：

- 当前高级配置字段较多，单纯折叠/展开无法解决“知道有这个配置但找不到在哪”的问题。
- 一键跳转可以把“滚动查找”改为“目标定位”，降低误配置概率和操作时间。
- 若跳转后仍需手动滚回顶部，体验仍然中断；持续导航闭环可显著降低来回滚动成本。
- 侧边目录符合文档型产品的操作习惯，适合承载不断增长的配置项数量，减少视觉迷失感。
- 固定操作栏与离开确认属于长表单的基本防误操作能力，能显著降低重复编辑成本。
- 跳转高亮可直接把“我跳到了哪里”可视化，减少视觉疲劳下的定位失败。
- 配置排序需遵循编辑心智：先定义“这是谁”（名称/描述/官网/密钥），再定义“怎么接入与路由”，最后处理“成本与稳定性”。
- 右侧配置区必须保持独立滚动，否则固定头尾会挤压有效编辑空间并导致长表单不可操作。

**替代方案**：

- 仅增加帮助文案，不提供跳转：信息可见但定位成本仍高。
- 将所有配置拆到多个弹窗：结构更重，交互负担更高且影响现有流程。

## Risks / Trade-offs

- [单实例并发计数与多实例不一致] → 本次范围内明确为单实例语义；后续如引入多实例，追加分布式并发控制。
- [新增并发过滤后可用候选减少导致更多降级] → 在 UI 与日志中展示并发满原因，并提供上限配置指导。
- [上游页重构范围较大影响回归面] → 分阶段交付：先数据与行为，再替换主列表视图；保留关键回归测试。
- [自动补全规则误判] → 使用能力映射白名单 + 预览框兜底，避免隐式修改不可见。

## Migration Plan

1. 扩展数据模型与 API（`max_concurrency`、官网地址、`last_used_at` 只读聚合字段、日志原因枚举扩展）。
2. 上线并发过滤与无熔断转移语义，确保运行时行为先稳定。
3. 上线日志时间线并发满原因展示，形成可观测闭环。
4. 重构上游管理页为卡片工作台，并替换 `created_at` 为 `last_used_at`。
5. 灰度验证：
   - 并发满时是否只转移不熔断；
   - endpoint 预览与实际请求地址一致；
   - 日志可追溯并发满转移链路。
6. 回滚策略：
   - 功能开关回退到旧列表视图；
   - 并发过滤逻辑可按配置关闭（等价“不限流”）。

## Open Questions

- 日志页并发满信息在紧凑视图中显示为徽标还是仅在详情面板显示？
- 上游工作台是否保留“经典表格视图”作为可切换兼容模式？
