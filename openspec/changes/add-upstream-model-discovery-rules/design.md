## Context

当前 AutoRouter 已经具备与模型相关的两类基础能力：

- 控制面已经可以在上游配置中保存 `allowed_models` 与 `model_redirects`，并通过管理端 CRUD 与表单进行维护。
- 运行时主链路已经基于路径能力与负载均衡完成上游选择，但主 proxy 路径并不会直接调用 `routeByModel()`，而是在选中上游后才应用 `resolveModelWithRedirects()`。

这意味着 issue #110 不是单纯补一个“从 `/v1/models` 拉数据”的功能，而是需要同时补齐三层能力：

1. 上游模型目录发现与缓存。
2. 管理端把目录转换为显式允许规则的能力。
3. 运行时在路径能力匹配之后、负载均衡选择之前，对显式模型规则做真正的候选过滤。

如果这三层不拆开，系统会出现“管理端看起来配置了模型白名单，但运行时仍然不会严格执行”的伪完成状态。

本次设计同时涉及数据模型、管理端表单与工作台、管理 API、模型发现服务、运行时路由决策与测试矩阵，因此需要独立 design 文档来固定语义边界。

### 当前与目标链路对比

```text
当前链路
══════════════════════════════════════════════════════════

request model
    │
    ▼
path capability matching
    │
    ▼
load balancer selects upstream
    │
    ▼
apply model_redirects on chosen upstream
    │
    ▼
forward downstream


目标链路
══════════════════════════════════════════════════════════

request model
    │
    ▼
path capability matching
    │
    ▼
explicit model rule filtering
(exact / regex / manual alias)
    │
    ▼
load balancer selects from eligible upstreams
    │
    ▼
apply resolved upstream model target
    │
    ▼
forward downstream
```

## Goals / Non-Goals

**Goals:**

- 为每个上游提供明确的模型发现配置与抓取状态记录。
- 将发现结果持久化为“目录缓存”，并保留来源、抓取时间、成功状态与失败原因。
- 允许管理员从目录缓存中选择模型并导入为显式允许规则。
- 扩展允许规则表达能力，支持精确模型名、正则模式与手工别名。
- 在运行时把显式模型规则纳入候选过滤，使其成为真实生效的准入约束，而不是仅停留在控制面展示。
- 在原生发现失败时提供 LiteLLM 推断候选目录作为辅助来源，并明确标注其非权威性。

**Non-Goals:**

- 不把 LiteLLM 候选目录视为上游权威返回，也不允许它在未导入为显式规则时直接改变运行时候选语义。
- 不在本次变更中引入自动定时同步、分布式抓取调度或后台任务编排系统。
- 不替换现有路径能力匹配链路；模型规则只在能力匹配之后补充候选过滤。
- 不重新引入基于 `provider_type` 的旧式路由语义。

## Decisions

### Decision 1: 将“模型目录缓存”和“显式允许规则”定义为两层不同数据

**Decision:**
目录缓存只表达“系统发现到了什么”，显式允许规则表达“该上游运行时允许什么”。从目录中导入是一个显式动作，而不是目录自动接管运行时决策。

**Rationale:**

- 目录数据存在时效性与权威性差异，特别是 LiteLLM fallback 只能视为推断候选。
- 运行时需要稳定、可审计的显式规则，不能直接把缓存目录当成执行契约。
- 这可以把“抓取失败/目录过期”和“运行时是否允许”分离，降低控制面与数据面耦合。

**Alternatives considered:**

- 直接把发现目录作为运行时白名单：实现更省事，但会让缓存新鲜度与错误抓取直接影响请求准入。
- 完全不持久化目录、仅允许临时查看：无法支撑导入、对比和失败回溯，运营价值不足。

### Decision 2: 允许规则统一建模为 exact / regex / alias 三类显式策略

**Decision:**
显式允许规则分为三类：

- exact：明确允许的模型名。
- regex：允许一组命名变体。
- alias：把管理员自定义模型名映射到上游目标模型名。

导入目录默认生成 exact 规则；手工补充可以新增 regex 与 alias 规则。

**Rationale:**

- issue 的核心痛点并不是只有“发现模型”，还包括企业命名、后缀变体与代理别名表达。
- 把三类策略拆开建模后，管理端展示、校验与运行时匹配顺序都更清晰。

**Alternatives considered:**

- 继续沿用 `allowed_models: string[]` 与 `model_redirects: Record<string, string>`：无法表达 regex，也无法区分人工 alias 与普通 redirect。
- 只支持 exact 与 regex，不支持 alias：仍然无法覆盖企业内部自定义名称到真实上游模型名的映射需求。

### Decision 3: 运行时在“路径能力匹配后、负载均衡前”执行显式模型规则过滤

**Decision:**
运行时先按路径能力构建候选集，然后对声明了显式模型规则的上游执行模型匹配过滤，仅保留 exact/regex/alias 命中的候选，再进入现有健康、熔断、配额、并发与权重选择。

**Rationale:**

- 这与现有主链路最兼容，不需要重做路径能力体系。
- 它能满足“显式白名单未命中就不应继续参与候选”的需求。
- 它避免了在已经选中单个 upstream 之后才发现模型不支持的语义错位。

**Alternatives considered:**

- 沿用 `routeByModel()` 的旁路逻辑：不能覆盖当前主 proxy 链路，容易继续造成控制面与运行时脱节。
- 在选中 upstream 之后再做严格拒绝：会让负载均衡与 failover 先基于错误候选工作，日志语义也更难解释。

### Decision 4: LiteLLM fallback 仅作为“可导入候选来源”而不是直接白名单

**Decision:**
当原生发现失败时，系统可以按上游类型读取 LiteLLM 候选目录，并将条目标记为 `inferred`。这些条目只能在管理端浏览与导入，不能在未导入前自动改变运行时允许结果。

**Rationale:**

- LiteLLM 提供的是整理后的兼容候选集，而不是当前上游真实返回。
- 明确限制使用方式后，既能提供兜底体验，也不会误导管理员把推断结果当成权威清单。

**Alternatives considered:**

- 把 LiteLLM 结果与原生发现合并为同级权威来源：会掩盖来源差异，增加误配风险。
- 完全不提供 LiteLLM fallback：会让非标准上游或原生发现失败场景回到纯手工输入，体验退化明显。

### Decision 5: 管理端用“发现区 + 规则区”双区块承载模型配置

**Decision:**
在上游表单或工作台中，将模型相关配置拆成两个显式区域：

- 模型发现区：配置发现方式、发现端点、查看最近抓取状态与目录来源。
- 模型规则区：管理 exact / regex / alias 规则，并支持从目录选择条目导入。

**Rationale:**

- “发现到了什么”和“实际允许什么”语义不同，混在一个文本域里会加剧误解。
- 这与控制面/数据面的分层一致，也便于后续扩展更多来源状态与导入动作。

**Alternatives considered:**

- 继续把所有模型配置塞进一个高级文本区：成本低，但难以展示来源、抓取状态与导入动作。
- 单独新增独立页面：信息更完整，但会让上游配置被拆散，不符合当前工作台式维护流。

## Risks / Trade-offs

- [显式规则与旧字段迁移语义不清] → 先保证旧 `allowed_models` / `model_redirects` 能被无损映射到新规则结构，再逐步开放 richer rule editor。
- [运行时执法点改变影响 failover 语义] → 将模型规则过滤放在负载均衡前，并补充日志字段说明候选是因模型规则被排除还是因健康/配额被排除。
- [原生发现接口差异大，抓取结果结构不统一] → 统一规范化为内部目录缓存结构，只暴露系统支持的通用元数据。
- [LiteLLM fallback 被误认为权威数据] → 在目录浏览与导入界面显式展示来源标签、抓取方式与“推断结果”警示文案。
- [管理端交互过重] → 将发现与规则区分层展示，只在需要时展开目录浏览与批量导入交互。

## Migration Plan

1. 为上游配置与模型目录缓存补充新的持久化结构，并保持旧配置兼容读取。
2. 在管理 API 与类型层引入 discovery config、catalog cache 与 rule set 的读写契约。
3. 在上游表单/工作台增加模型发现区与模型规则区，并提供目录浏览与导入流程。
4. 在 proxy 主链路中引入显式模型规则过滤，将其接入现有路径能力匹配后的候选构建逻辑。
5. 增补 CRUD、表单、模型发现、proxy 路由与日志观测测试，确认 strict rule filtering 与 fallback 标记行为可验证。
6. 如需回滚，保留旧字段解析与旧表单兼容分支，先回退运行时过滤，再回退管理端 richer rule editor。

## Open Questions

- alias 规则的匹配结果是否只允许映射到当前上游已知目录中的模型，还是允许管理员映射到未被发现但人工确认存在的目标模型。
- 目录缓存是否需要保留“最后一次成功结果”与“最近一次失败状态”两套并行状态，以避免抓取失败时界面完全丢失上次目录。
- 对 Gemini native 这类返回结构不同的发现端点，管理端是否需要展示能力型元数据，还是只保留统一模型名视图。
