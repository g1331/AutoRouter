## Context

issue #142 的四个后端变更已交付 CLIProxyAPI 集成的全部 Admin API。本变更交付配套的前端管理界面，使管理员能够通过界面而非直接调用 API 来使用这套集成。

后端 API 面（均位于 `/api/admin/cliproxy`）已固定：实例的列表、创建、读取、更新、删除与连通性检测；账号的列表、同步、字段编辑、启停；OAuth 登录的发起与状态轮询；池上游与单账号上游的创建。这些接口不在本变更中改动。

AutoRouter 前端模式已成熟。dashboard 页面均为客户端组件，以 `Topbar` 开头，使用 `Card` 承载过滤区与表格，创建编辑与删除通过 `Dialog` 完成，数据获取集中在 `src/hooks/use-<feature>.ts`，表单使用 `react-hook-form` 配合 `zod`，提示使用 `sonner`。系统类页面位于 `(dashboard)/system/<name>/`。本变更严格复用这套模式。

## Goals / Non-Goals

**Goals:**

在左侧菜单交付 CLIProxyAPI 入口与对应页面。交付实例管理、OAuth 账号管理、OAuth 登录流程、CLI OAuth 上游创建四类界面能力。所有界面调用既有 Admin API，复用既有前端组件与模式。补齐中英文文案。按项目 TDD 规范为新增 hooks 与关键组件补测试。

**Non-Goals:**

不修改任何后端 API、服务、数据库 schema。不新增后端能力。不引入新的前端基础设施或新的 UI 框架。不交付 sidecar 部署相关界面，部署由 `cliproxy-sidecar-deployment` 变更的文档覆盖。不提供 OAuth 登录的二维码展示，首版以可点击链接与复制按钮为主。

## Decisions

UX 关键决策经与 Codex 商定，结论如下。

### 决策一：单页承载，实例表格在上，账号面板在下

CLIProxyAPI 的数据是两层结构：实例，以及隶属于实例的账号与上游。界面采用单一页面 `(dashboard)/system/cliproxy`，与现有全部 system 页面的单层结构一致，不引入 `[id]` 子路由。

页面上方是实例表格，承载实例的增删改查与连通性检测。选中某个实例后，该实例的 OAuth 账号列表在同页实例表格下方的面板中展示。实例数量通常少（一到数个），适合作为轻量入口列表；账号数量可能较多，下方面板提供独立的滚动空间与完整操作列。

Codex 的原始建议是以右侧划出式面板（slide-over）承载账号列表。当前 `src/components/ui/` 没有 slide-over 或 Sheet 组件，为避免为单一功能引入新的基础组件，账号列表改以页面内联面板呈现，承载方式从 slide-over 调整为内联 `Card` 面板，单页结构与「账号有独立滚动空间」的意图保持不变。

### 决策二：OAuth 登录以弹窗承载完整流程

OAuth 登录是强状态的短时任务。发起登录的接口返回授权地址与轮询用的会话标识，需轮询登录状态接口直到完成。

需要说明的是，issue #142 原文提到展示 device code 与过期时间，但 `cliproxy-oauth-account-management` 变更交付的发起登录接口实际只返回授权地址与会话标识，不返回 device code，也不返回过期时间。本界面按实际接口契约实现：不展示 device code，不展示服务端过期倒计时。

界面以 `Dialog` 弹窗承载完整流程，与既有创建编辑弹窗模式一致，并与账号表格隔离。弹窗内提供「在新标签页打开授权地址」按钮与复制授权地址按钮，轮询状态以加载指示与文字提示呈现。

轮询间隔为 3 秒。由于接口不返回过期时间，以客户端固定超时上限作为硬性截止。登录成功时关闭弹窗、刷新账号列表并以成功提示反馈。失败或达到客户端超时上限时停止轮询，展示明确错误与重新发起登录的入口。用户主动关闭弹窗时停止轮询。

### 决策三：上游创建入口按作用对象就近放置

一键创建池上游的作用对象是整个实例，入口放在实例行的操作菜单，与连通性检测、编辑、删除同层。将单个账号固定映射为上游的作用对象是具体的 OAuth 账号，入口放在账号行的操作菜单。两类入口的放置与其作用范围一致，避免混淆池上游与单账号上游。

两类创建操作均经确认弹窗完成，防止误触重复创建。

### 决策四：单一变更，分五个实现阶段

后端 API 已完整可用，本变更全部工作围绕同一个管理页面展开。拆成多个变更会产生只有入口没有功能的半成品页面，且菜单注册、页面路由、i18n 等基础层需反复修改。因此保持单一变更，内部分五个阶段推进，每阶段独立提交以控制评审范围：菜单与路由与实例界面、账号列表与账号管理、OAuth 登录流程、上游创建、国际化与收尾。

## Risks / Trade-offs

OAuth 登录轮询可能因接口持续不返回完成状态而长时间挂起 → 以接口返回的过期时间为硬性截止，到期即停止轮询并提示，避免无限轮询。

账号同步与 OAuth 登录依赖 CLIProxyAPI 在线 → 实例未通过连通性检测或 CLIProxyAPI 不可达时，相关操作会失败，界面以既有 `sonner` 错误提示呈现可理解的失败信息，与既有页面的错误处理方式一致。

单页同时承载实例与账号两层数据 → 通过「选中实例」状态切换下方账号面板，TanStack Query 以实例 id 为查询键区分账号数据，切换实例时自动重新获取。

## Migration Plan

本变更为纯前端新增，无数据库迁移。新增菜单项与页面对既有页面零影响。新增 hooks 与组件不被既有代码引用。回滚方式为移除菜单项、页面与新增组件文件。

## Open Questions

无。四项 UX 决策已确定。组件拆分粒度与表单字段校验细节在 tasks 阶段随实现按既有页面模式落实。
